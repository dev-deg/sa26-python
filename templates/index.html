Securing Applications Lab — XXE & XSS

Intentionally vulnerable endpoints for educational use. Compare each attack against its secure counterpart.

🛡

XSS — Cross-Site Scripting

Injecting malicious JavaScript into pages viewed by other users

XSS occurs when a server reflects user-supplied input back into HTML without escaping it. The attacker does not need server access — they exploit the application’s failure to treat user input as data rather than code.

🚫 How the attack works

  1. Attacker finds a URL param or input field reflected in the page.
  2. Injects a <script> tag or event handler (onerror, onload).
  3. Server echoes the payload unescaped into the HTML response.
  4. Victim’s browser executes the injected script.
  5. Script steals cookies, exfiltrates data, or hijacks the session.

🟢 Why cookies matter here

  1. Both demo pages set fake session_id and user_role cookies.
  2. The vulnerable page sets them without HttpOnly — JS can read them.
  3. The secure page sets them with HttpOnly=True — JS is blind.
  4. alert(document.cookie) shows the difference instantly.

Demo endpoints

EndpointTypeWhat to observe
/vuln/xss?name=World VULNERABLE Raw input injected into HTML — scripts execute
/secure/xss?name=World SECURE Jinja2 auto-escaping applied — tags rendered as text

Try it — paste into browser address bar

browser 
http://localhost:8000/vuln/xss?name=<script>alert(document.cookie)</script>
browser — secure page (same payload, no alert) 
http://localhost:8000/secure/xss?name=<script>alert(document.cookie)</script>
curl -G "http://localhost:8000/vuln/xss" --data-urlencode "name=<script>alert('XSS')</script>"
Invoke-WebRequest -Uri "http://localhost:8000/vuln/xss?name=%3Cscript%3Ealert('XSS')%3C%2Fscript%3E" | Select-Object -Expand Content

🔐 Defense techniques

  • Escape all output — Python: html.escape(); Jinja2 auto-escapes by default.
  • Content-Security-Policy header restricts which scripts browsers will run.
  • HttpOnly cookies make session tokens invisible to JavaScript.
  • Never use string concatenation to build HTML — use a template engine.

XXE — XML External Entity

Abusing XML parsers to read local files, perform SSRF, or crash servers

XXE occurs when an XML parser is configured to process external entity references declared in a DTD. An attacker can point an entity at file:///etc/passwd and the parser substitutes the file contents directly into the document — which the application then returns.

🚫 How the attack works

  1. Attacker crafts XML with a DOCTYPE declaring an external entity.
  2. Entity SYSTEM URI points to a local file or internal URL.
  3. Vulnerable parser resolves the entity, reading the file.
  4. File contents are substituted into the document.
  5. Application returns the parsed result — leaking the file.

🚨 Three attack variants

  1. File read — exfiltrate /etc/passwd, SSH keys, config files.
  2. Secret theft — read .env to steal JWT secrets and DB credentials.
  3. Billion laughs — recursive entity expansion causes ~3 GB RAM spike (DoS).

Try it — cross-platform (Mac, Windows, Linux)

The ?demo=secrets flag makes the server build a payload targeting demo_files/fake_secrets.env with the correct absolute path for your OS. No real credentials are exposed. 

curl -X POST "http://localhost:8000/vuln/xxe?demo=secrets"
Invoke-RestMethod -Method Post -Uri "http://localhost:8000/vuln/xxe?demo=secrets"
curl -X POST "http://localhost:8000/secure/xxe?demo=secrets"
Invoke-RestMethod -Method Post -Uri "http://localhost:8000/secure/xxe?demo=secrets"

Try it — billion laughs DoS (all platforms)

Sends a recursive entity payload. Both endpoints block it — but for different reasons, which is the interesting part. The secure endpoint is stopped by defusedxml at the Python layer. The vulnerable endpoint is stopped by libxml2's built-in C-level amplification limit — a defence that many older parsers (PHP SimpleXML, Java DocumentBuilder) do not have, making them truly exploitable. 

curl -X POST "http://localhost:8000/secure/xxe?demo=billion_laughs"
Invoke-RestMethod -Method Post -Uri "http://localhost:8000/secure/xxe?demo=billion_laughs"
curl -X POST "http://localhost:8000/vuln/xxe?demo=billion_laughs"
Invoke-RestMethod -Method Post -Uri "http://localhost:8000/vuln/xxe?demo=billion_laughs"

Try it — static XML files (Mac / Linux only)

These target Unix paths (/etc/passwd). On Windows, edit the SYSTEM URI in the XML file to file:///C:/Windows/System32/drivers/etc/hosts. 

curl -X POST http://localhost:8000/vuln/xxe -H "Content-Type: application/xml" --data-binary @attack_files/xxe_basic.xml
Invoke-RestMethod -Method Post -Uri "http://localhost:8000/vuln/xxe" -ContentType "application/xml" -InFile "attack_files\xxe_basic.xml"
curl -X POST http://localhost:8000/secure/xxe -H "Content-Type: application/xml" --data-binary @attack_files/xxe_basic.xml
Invoke-RestMethod -Method Post -Uri "http://localhost:8000/secure/xxe" -ContentType "application/xml" -InFile "attack_files\xxe_basic.xml"

🔐 Defense techniques

  • Use defusedxml — drop-in stdlib replacement; blocks all XXE and DoS variants.
  • Disable DTDs in lxml — XMLParser(resolve_entities=False, load_dtd=False).
  • Prefer JSON over XML for APIs where XML isn’t required.
  • Validate structure before parsing — reject unexpected DOCTYPEs at the application layer.
